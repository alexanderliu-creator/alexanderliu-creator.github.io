PKU-OS and Virtual Security-操作系统简介

本文最后更新于：3 天前

操作系统简介

操作系统主流架构

  • 分类：
    • 微内核：只在内核里面实现最基本的功能。（小而美，模块化精细，相互之间可以调用）
    • 宏内核：在内核里面实现所有的功能。（大而全，模块化程度不够好，耦合性高）

Kernel就是这里的内核功能。

  • 微内核示例：
    • User mode: Application Program, File System, Device Driver
    • Kernel mode: Interprocess communication, memory management, CPU scheduling
    • Hardware

Why MicroKernel: 可证安全：足够小，方便证明！Macro太大了，但是Micro足够小，每个独立功能模块都能被证明是安全的。

  • 微内核特点：

    • 足够小的内核。
    • C/S架构，User mode有很多S，供C调用。
    • 策略与机制分离：策略在User Mode，实现在Kernel Mode，方便策略拓展。
    • 面向对象的技术，较好的结构性和抽象。

  • 微内核系统有：

    • Minix
    • WindowsNT/XP/Barrelfish/10
    • Mac OS

  • Multi core / Many core

  • 宏内核系统有：Linux/UNIX

  • 微内核好：可伸缩性好、可移植性好、实时性好、安全可靠性高、支持分布式系统、真正面向对象的操作系统

操作系统简介

  • 操作系统技术发展：网络计算、多核计算、可信计算、虚拟化计算、移动计算、云计算

OS本质就是管理硬件 or 下层系统，对上层提供简单接口，好用。

  • 系统Focus不同，根据不同的业务场景，也许需要不同的能力增强与开发，因使用环境而变。（例如大型机房操作系统的能力和个人电脑的能力就不一样，操作系统的设计也就不一样捏）
  • 操作系统功能：CPU, Storage, IO, File System, User Interface
  • 特性：并发性，共享性，异步性
  • 操作系统如何引导和控制计算机：
    • 上电到操作系统运行：外部存储器 -> OS -> RAM，加载操作系统就是这么走的。
    • 流程概览：
      • Step1: 上电：加电运行，第一个状态是CPU处理机生产商的硬件电路决定的，x86设置指令寄存器为0ffff: 0000。会从这个指令寄存器跳转到BIOS（ROM）软件的入口。（信任根就可以放在这里）
      • Step2: BIOS自检和加载引导扇区。进行自检，加载引导扇区。
      • Step3: 把操作系统引导扇区的内容加载到内存中来，并且跳转到引导程序的第一条指令。
      • Step4: 把选择的操作系统镜像，加载进内存（引导）。
      • Step5: 运行操作系统

从开机加电到执行main之前的过程

  • 从开机到main函数的执行分三步完成，目的是实现从启动盘加载操作系统程序，完成执行main函数所需要的准备工作。
    • 第一步， 启动BIOS，准备实模式下的中断向量表和中断服务程序;
    • 第二步，从启动盘加载操作系统程序到内存，加载操作系统程序的工作就是利用。第一步中准备的中断服务程序实现的。
    • 第三步，为执行32位的main函数做过渡工作。将详细分析这三步在计算机中是如何完成的，以及每一步在内存中都做了些什么。

启动BIOS

BIOS启动原理

CPU只能计算RAM中的数据，但是一开机，RAM是空的。如果要运行操作系统，就必须先把操作系统程序加载到RAM中。

  • RAM: 加电后可以读写，断电后数据消失。
  • BIOS: BIOS ROM芯片中装有“系统设置程序主要用来设置CMOSRAM中的各项参数。工控机开机时按某个键，这个程序就可进入设置状态，并提供良好的界面。BIOS负责完成加载软盘中操作系统的任务。
  • ROM: 只读存储器，和硬盘差不多，断电也能存储数据。
  • 开机的时候，内存里啥都没有，如何BIOS？Secret is: 0xFFFF0，默认地址。软件不可能执行BIOS，只能硬件来实现。关键的地方是：将CPU的硬件逻辑设计为加电瞬间强行设置CS和IP，指向0xFFFF0，0xFFFF0就是BIOS的地址范围。（纯硬件完成），如果没有可执行的代码，就done了。

BIOS在内存中加载中断向量表和中断服务程序

BIOS被固化在ROM中，原理大致相似。BIOS在内存中加载中断向量表和中断服务程序对于boot十分重要。

  • BIOS在内存最开始的位置，1kB内存构建中断向量表，紧挨着256 bytes构建BIOS数据区。并且在后面加载了若干中断程序。

加载操作系统内核并准备保护模式

Bootsect

  • 中断向量表( Interrupt Vector Table): 实模式中断机制的重要组成部分，记录所有中断号对应的中断服务程序的内存地址。
  • 中断服务(Interrupt Service)程序: 通过中断向量表的索引对中断进行响应服务，是一些具有特定功能的程序。
  • BIOS的工作是：找到软盘 并 加载第一扇区。这就是引导程序bootsect.s，然后再执行引导程序，将后面的数据加载进入内存昂。

Setup

  • BIOS已经把bootsect，也就是引导程序放入内存，作用就是
    1. 完成内存规划，把内存管理好。
    2. 把自己在内存中移一下位置，把低地址空间的位置让出来。

System Module

  • bootsec借着BIOS，将system加载进系统。(int 0x13)

进入32位模式，准备调用Main函数

关中断 & system移动到0x00000

  • 关中断(cli) 和开中断(sti) 操作将在操作系统代码中频繁出现，其意义深刻。慢慢的你会发现，cli、 sti 总是在一个完整操作过程的两头出现，目的是避免中断在此期间的介入。接下来的代码将为操作系统进入保护模式做准备。此处即将进行实模式下中断向量表和保护模式下中断描述符表(IDT)的交接工作。试想，如果没有cli，又恰好发生中断，如用户不小心碰了一下键盘，中断就要切进来，就不得不面对实模式的中断机制已经废除、保护模式的中断机制尚未完成的尴尬局面，结果就是系统崩溃。cli、 sti 保证了这个过程中，IDT能够完整创建，以避免不可预料中断的进入造成IDT创建不完整或新老中断机制混用。甚至可以理解为cli、sti是为了保护一个新的计算机生命的完整而创建的。
  • 将内核代码复制到内存0x00000处。

  • 0x00000这个位置原来存放着由BIOS建立的中断向量表及BIOS数据区。这个复制动作将BIOS中断向量表和BIOS数据区完全覆盖，使它们不复存在。直到新的中断服务体系构建完毕之前，操作系统不再具备响应并处理中断的能力。这样做能取得“一箭三雕”的效果:
    • 废除BIOS的中断向量表，等同于废除了BIOS提供的实模式下的中断服务程序。
    • 收回刚刚结束使用寿命的程序所占内存空间。
    • 让内核代码占据内存物理地址最开始的、天然的、有利的位置。

废除了BIOS的中断向量表，也就是废除了16位的中断机制。操作系统是不能没有中断的，对外设的使用、系统调用、进程调度都离不开中断。Linux 操作系统是32位的现代操作系统，16位的中断机制对32位的操作系统而言，显然是不合适的，这也是废除16位中断机制的根本原因。为了建立32位的操作系统，我们不但要“破旧”，还要“立新”一建 立新的中断机制。

设置中断描述符表和全局描述符表

  • Setup对IDTR和GDTR进行初始化
  • 32位的中断机制和16位的中断机制，在原理上有比较大的差别。最明显的是16位的中断机制用的是中断向量表，中断向量表的起始位置在0x00000处，这个位置是固定的: 32位的中断机制用的是中断描述符表(IDT)， 位置是不固定的，可以由操作系统的设计者根据设计要求灵活安排，由IDTR来锁定其位置。GDT是保护模式下管理段描述符的数据结构，对操作系统自身的运行以及管理、调度进
    程有重大意义，后面的章节会有详细讲解。因为，此时此刻内核尚未真正运行起来，还没有进程，所以现在创建的GDT第一项为
    空，第二项为内核代码段描述符，第三项为内核数据段描述符，其余项皆为空。IDT虽然已经设置，实为一张空表，原因是目前已关中断，无需调用中断服务程序。
  • 此处反映的是数据“够用即得”的思想。创建这两个表的过程可理解为是分两步进行的:
    • 在设计内核代码时，已经将两个表写好，并且把需要的数据也写好。
    • 将专用寄存器(IDTR、GDTR)指向表。此处的数据区域是在内核源代码中设定、编译并直接加载至内存形成的一块数据区域。专用寄存器的指向由程序中的lidt和lgdt指令完成，具体操作见图1-18。
  • 值得一提的是，在内存中做出数据的方法有两种:1)划分一块内存区域并初始化数据，“看住”这块内存区域，使之能被找到;2)由代码做出数据，如用push代码压栈，“做出”数据。此处采用的是第一种方法。
