语言进阶-并发编程

并发 vs 并行

Go可以充分发挥多核优势，高效运行

协程

Goroutine

  • Goroutine:

线程：内核态，线程跑多个协程，栈KB级别。（内核态，切换成本也高）

协程：用户态，轻量级线程，栈MB级别。（创建和调度由Go来完成昂！！！）

  • 开启协程：

Channel

  • 协程通信：

通过通信共享内存，而不是通过共享内存来通信

通过管道(pipe)，进行数据共享，很优雅。

但是通过共享内存（可能还要加锁），反而复杂了，有可能产生死锁，data race等问题昂！！！

  • Channel:

make(chan int) -> 无缓冲通道（同步通道）

make(chan int, 2) -> 有缓冲通道（通道中能存放2个元素）

本质还是Producer-Consumer模型昂！！！

带缓冲的Channel，可以一定程度上解决生产者和消费者速度不一致的问题。

Sync

Lock

  • 并发安全Lock:

没有死锁，但是共享变量由于没有很好的用锁保护，产生了data race的问题。-> 并发安全问题！！！

WaitGroup

WaitGroup暴露的方法只有三个昂，Add(), Done()和Wait()，内部就是维护了一个计数器昂！！！

依赖管理

背景

依赖管理演进

GoPath -> Go Vendor -> Go Module

  • 不同环境（项目）依赖的版本不同
  • 控制依赖库的版本

GOPATH

环境变量：$GOPATH

项目代码直接依赖src下的代码

go get下载最新的版本的包也是到src目录下

逻辑：所有的代码都在src下！！！

  • 问题：

B升级后，没有做版本的兼容，A和B依赖于package的不同版本。GOPATH无法实现package的多版本控制。

Go Vendor

项目目录下增加vendor文件，所有依赖包副本形式放在$ProjectRoot/vendor下

依赖寻址方式：vendor => GOPATH

通过每个项目引入一份依赖的副本，解决了多个项目需要同一个package依赖的冲突问题。

  • 问题：

无法控制依赖的版本

更新项目可能导致出现依赖冲突，导致编译出错。

Go Module

  • 通过go.mod文件管理包的版本
  • 通过go get/go mod指令工具管理依赖包

终极目标：定义版本规则和管理项目依赖关系。

  • 依赖管理三要素：
    1. 配置文件，依赖描述 — go.mod
    2. 中心仓库管理依赖库 — Proxy
    3. 本地工具 — go get/mod

和Java里面的Maven类似的昂！！！

  • 结构分析：

  • 依赖配置 - version

    • 语义化版本：Major.Minor.Patch(e.g. v1.0.2)
    • 基于commit的伪版本：版本前缀（语义版本）-时间戳-哈希校验码

  • Indirect: 间接依赖，A -> B -> C。A -> B是直接依赖，A -> C是间接依赖（Indirect）

  • Incompatible（兼容之前的仓库）：

    • 主版本2+模块会在路径增加/vN后缀。
    • 没有go.mod文件并且主版本2+的依赖，会+incompatible

依赖配置

编译时所用的C的版本是1.4，会选择最低的兼容版本昂！！！

依赖分发-Proxy

  • 无法保证构建稳定性（软件版本变动多）
  • 无法保证依赖可用性（被删掉了）
  • 增加第三方压力

Proxy就能够帮助我们去保持依赖的稳定性昂！！！

  • 配置：GOPROXY=”https://proxy1.cn, https//proxy2.cn, direct”

  • 例子：

  • 很多开源包都可以帮忙测试的昂

覆盖率

go test judgment_test.go judgment.go --cover

--cover就能得到覆盖率昂！！！

测试中运行了的行，和所有的行的比例，就是覆盖率昂！

  • 增多测试用例，把未覆盖的行给覆盖了，那么覆盖率就up了昂！

  • Tips:

    • 一般50%到60%就差不多了，较高的覆盖率80%。（覆盖率提高是有成本的昂）
    • 测试分支相互独立，全面覆盖。
    • 测试单元粒度足够小，函数单一职责。
